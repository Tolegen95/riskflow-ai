🎓 0. Научная рамка исследования
ОбъектПроцессы CSIRT и управление рисками информационной безопасности при реагировании на инциденты.
ПредметМетод процессно-ориентированной оценки пробелов безопасности с учётом угроз, уязвимостей, влияния на реагирование и эффективности мер контроля.
ЦельРазработать и апробировать программно-методический комплекс для оценки, согласования и контроля обработки рисков CSIRT-процессов.
АртефактРабочий прототип RiskFlow AI: реестр пробелов, BPMN-подобная модель процесса, количественная оценка риска, workflow, RBAC, audit log, compliance и отчётность.
Формула научной новизны:
предложена связка CSIRT-процесс → шаг BPMN → пробел безопасности → риск → мера обработки → свидетельство compliance, позволяющая оценивать не только отдельные активы, но и организационные слабые места процесса реагирования.

Проверяемые положения: методика повышает полноту фиксации пробелов, делает оценки воспроизводимыми, связывает риск с ответственными ролями и позволяет контролировать жизненный цикл обработки риска.

Ограничения: текущий модуль рекомендаций является rule-based, демонстрационная база не заменяет промышленную апробацию, а BPMN-модуль хранит кастомный JSON и не заявляется как полный BPMN 2.0 engine.

🏗️ 2. Архитектура системы (обзорная схема)

Система построена на трёхзвенной архитектуре: Presentation Layer (Jinja2 + Tailwind CSS), Application Layer (Flask, Python 3.13), Data Layer (SQLite, 25 таблиц). Каждый модуль изолирован и взаимодействует через маршруты Flask — это позволяет тестировать каждый модуль независимо.

📐 3. Математическая модель оценки рисков
# Шаг 1: Потенциал реализации угрозы P_event = max(P + V − 1, 0) # P ∈ [0,3] — Вероятность инцидента (1 низкая, 2 средняя, 3 высокая) # V ∈ [0,3] — Выраженность уязвимости / организационного пробела # Шаг 2: Начальный уровень риска R_initial = P_event × I # I ∈ [0,9] — Влияние на реагирование # Шаг 3: Остаточный риск R_residual = R_initial × (1 − CE) # CE ∈ [0,1] — Эффективность контроля # Шаг 4: Снижение риска ΔR = R_initial − R_residual
Классификация уровня риска (R_residual):
Диапазон R_residualУровеньДействие
0 — 3.9НизкийКонтрольный мониторинг
4.0 — 6.9СреднийПлан обработки, 30 дней
7.0 — ∞ВысокийНемедленное реагирование
Приоритет обработки (score = ценность + снижение + стоимость):
score ≥ 8 → HIGH · score ≥ 5 → MEDIUM · иначе → LOW

Формула адаптирована для процессного анализа CSIRT: параметр V отражает выраженность организационного или технического пробела, а CE показывает долю риска, компенсируемую текущими мерами контроля. Числовой показатель риска хранится как risk_level, а текстовая категория управления — как risk_category.

🧪 9. План экспериментальной валидации

Для докторской диссертации программный прототип должен быть подтверждён не только тестами кода, но и экспериментом с экспертами. Ниже приведён минимальный дизайн апробации, который можно выполнить до защиты и включить в главу с результатами исследования.

Элемент экспериментаРекомендуемое содержаниеОжидаемый результат
Участники 5-7 специалистов: CSIRT-аналитики, риск-менеджеры, аудиторы ИБ или преподаватели по ИБ. Для каждого участника фиксируются стаж, роль и опыт работы с инцидентами. Экспертная база для оценки применимости методики.
Сценарий Один демонстрационный процесс, например обработка фишингового инцидента или DDoS. Участники сначала оценивают пробелы вручную, затем выполняют тот же сценарий в RiskFlow AI. Сравнение ручного и автоматизированного подхода.
Метрики качества Полнота выявления пробелов, согласованность оценок P/V/I, время заполнения реестра, число ошибок в назначении ответственных, полнота evidence/compliance. Количественное доказательство пользы процессной методики.
Метрики UX SUS-анкета или короткая 5-балльная анкета: понятность шкал, удобство формы, полезность BPMN-связи, доверие к рекомендациям, пригодность отчёта. Средняя оценка удобства и список замечаний экспертов.
Критерии успешности Снижение времени анализа не менее чем на 20%; рост полноты выявления пробелов; SUS ≥ 70 или средняя экспертная оценка ≥ 4 из 5; отсутствие критичных замечаний к логике расчёта. Формальные критерии принятия методики.
Артефакты для приложения Анкета, обезличенные ответы, таблица сравнения ручного и автоматизированного анализа, скриншоты маршрута, CSV/ZIP-отчёт, протокол тестового прогона. Доказательная база для приложения к диссертации.
Рекомендуемая формулировка результата: экспериментальная апробация должна показать, что предложенная методика повышает воспроизводимость оценки рисков CSIRT-процессов и связывает выявленные пробелы с ответственными ролями, мерами обработки и evidence по требованиям стандартов.

Пошаговый пример использования по ролям

Проблема, которую решает пример: в CSIRT отсутствует единый контролируемый процесс обработки фишингового инцидента. Из-за этого triage выполняется по-разному разными аналитиками, критерии эскалации не формализованы, ответственные не назначены, evidence по выполненным мерам не собирается, а руководитель не видит остаточный риск.

Ниже приведён сценарий, который можно буквально повторить на демонстрации. Он показывает путь одной проблемы от создания процесса до аудиторской проверки.

Демо-кейсЧто должно получиться
«Нет формальных критериев эскалации фишингового инцидента на этапе triage» Создан процесс, зафиксирован пробел, рассчитан остаточный риск, назначена мера обработки, выполнен workflow, приложен evidence, аудитор видит полную историю.

1. Admin: подготовить демонстрационный процесс

ШагЧто нажатьЧто заполнить / проверитьЧто показать комиссии
1Войти: admin / admin123Открыть dashboard и убедиться, что роль admin отображается в интерфейсе.Администратор видит все разделы системы.
2Пользователи → ДобавитьСоздать роли для демонстрации: risk_manager, process_owner, expert, auditor.RBAC: у каждой роли отдельный пользователь.
3Организации → ДобавитьНазвание: «CSIRT умного города», отрасль: «Smart City / Security».Все процессы привязываются к организации.
4Процессы → ДобавитьНазвание: «Обработка фишингового инцидента», владелец: process_owner.Появился процесс, за который отвечает конкретный владелец.
5Процесс → Добавить подпроцессДобавить шаги: «Получение сообщения», «Triage», «Классификация», «Эскалация», «Containment», «Recovery», «Post-review».Процесс разложен на шаги, где можно искать пробелы.

2. Process Owner: создать пробел безопасности

ШагЧто нажатьЧто заполнить / проверитьЧто показать комиссии
1Войти: process_owner / po_passОткрыть Мои задачи или Процессы.Владелец видит только свои процессы.
2Процессы → Обработка фишингового инцидентаОткрыть карточку процесса и найти подпроцесс «Triage».Пробел привязан к конкретному шагу, а не к процессу в целом.

4Заполнить описание пробелаТекст: «Нет формальных критериев эскалации фишингового инцидента».Комиссия видит понятную управленческую проблему.
5Заполнить оценкуP=2.5, V=2.0, I=3.0, CE=0.2, confidence: Medium.Система автоматически рассчитывает остаточный риск и категорию.
6Назначить ответственныхRisk owner: process_owner; mitigation owner: process_owner; due date: ближайшие 30 дней.У риска появляется владелец и срок.
7Нажать workflow-действие submitКомментарий: «Передаю на проверку риск-менеджеру».Статус меняется с Draft на In Review.

3. Expert: проверить корректность оценки

ШагЧто нажатьЧто заполнить / проверитьЧто показать комиссии
1Войти: expert / ex_passОткрыть Реестр пробелов или Мои задачи.Эксперт видит риск, который ожидает проверки.
2Открыть пробел по triageПроверить P/V/I/CE: вероятность, уязвимость, влияние, эффективность контроля.Показывается, что оценка не принимается без экспертного просмотра.
3Открыть МетодологияСверить шкалы: P и V от 0 до 3, I от 0 до 9, CE от 0 до 1.Методика оценки формализована.
4Проверить rule-based рекомендациюНапример: усилить контроль, назначить меру, контролировать срок.Система не просто хранит риск, а предлагает управленческое действие.
5Добавить evidence или комментарийНапример: «В регламенте triage отсутствует таблица критериев эскалации».Появляется доказательство существования пробела.

4. Risk Manager: принять решение и назначить обработку

ШагЧто нажатьЧто заполнить / проверитьЧто показать комиссии
1Войти: risk_manager / rm_passОткрыть Workflow рисков.Risk manager видит риски, ожидающие решения.
2Открыть риск в статусе In ReviewПроверить residual risk, категорию, владельца, due date, evidence.Решение принимается на основании полных данных.
3Нажать approveКомментарий: «Оценка подтверждена, требуется формализация критериев эскалации».Статус меняется на Approved, запись попадает в историю workflow.
4Рекомендации → ДобавитьМера: «Разработать и утвердить матрицу эскалации фишинговых инцидентов»; тип: Mitigate; progress: 0%.У риска появляется конкретный план снижения.
5Нажать startКомментарий: «Мера передана владельцу процесса».Статус становится In Progress.
6После выполнения меры нажать closeПроверить evidence и фактическое снижение residual risk.Риск закрыт, история решений сохранена.

5. Auditor: проверить, что риск обработан корректно

ШагЧто нажатьЧто заполнить / проверитьЧто показать комиссии
1Войти: auditor / aud_passОткрыть Реестр пробелов.Аудитор видит данные без права изменения.
2Отфильтровать риски по статусу/уровнюНайти риск по triage и проверить его статус.Можно быстро найти критичные или незакрытые пробелы.
3Открыть Workflow рисковПроверить цепочку: Draft → In Review → Approved → In Progress → Closed.Понятно, кто и когда принимал решения.
4Открыть ComplianceПроверить, есть ли evidence по мере обработки.Риск связан с подтверждением выполнения требований.
5Открыть Отчёт по процессу или экспорт CSVСохранить результат проверки для приложения к диссертации или аудиту.Система формирует воспроизводимый отчет.
Итог примера: проблема «нет критериев эскалации фишингового инцидента» проходит полный цикл: выявление → экспертная оценка → утверждение → мера обработки → evidence → закрытие → аудит. В результате CSIRT получает не просто список проблем, а управляемый и проверяемый процесс снижения риска.

👥А.2 Роли пользователей и функциональные возможности

Система реализует модель управления доступом на основе ролей (RBAC — Role-Based Access Control). Определены пять ролей, образующих иерархию привилегий от максимальных (admin) до минимальных (auditor).

👑
Администратор (admin) — полный доступ без ограничений
✓ Доступно:
  • Управление пользователями: создание, редактирование, удаление учётных записей, назначение ролей
  • Управление организациями: CRUD-операции над companies
  • Управление экспертами: добавление, редактирование профилей экспертов
  • Управление процессами: создание, редактирование, удаление процессов и подпроцессов
  • BPMN-редактор: создание и изменение диаграмм процессов
  • Управление пробелами (рисками): все CRUD-операции, удаление
  • Workflow: все переходы (submit, approve, start, close, return)
  • Управление рекомендациями: все операции, удаление
  • Качество данных: просмотр отчётов об аномалиях в БД
  • Журнал аудита: просмотр всех событий системы
  • Аналитика: все отчёты и визуализации
  • Соответствие стандартам: управление свидетельствами
Технические ограничения:
  • Нет ограничений по видимости данных — видит все процессы и риски всех организаций
  • Декораторы: @admin_required, @risk_editor_required, @login_required
Сценарий использования:

Системный администратор или руководитель подразделения ИБ. Настраивает систему, заводит пользователей, контролирует качество данных, просматривает все события через журнал аудита.

📋
Менеджер рисков (risk_manager) — управление реестром и workflow
✓ Доступно:
  • Просмотр всех процессов и пробелов (без ограничений по владельцу)

  • Редактирование любых пробелов (если создал или назначен)

  • Редактирование рекомендаций
  • Все аналитические страницы: дашборд, workflow, матрица ответственности, compliance, операционная аналитика

✗ Недоступно:

  • Управление экспертами (/experts)

Сценарий использования:

Специалист по управлению рисками CSIRT. Ведёт реестр пробелов, утверждает или возвращает риски, контролирует исполнение рекомендаций, формирует отчёты для руководства.

🏢

✓ Доступно:

  • Журнал аудита, качество данных

  • Все workflow-переходы
  • Всё администрирование (пользователи, эксперты, процессы)
Сценарий использования:

Привлечённый специалист или внутренний аналитик. Просматривает назначенные задачи, изучает реестр пробелов, консультирует процесс-оунеров по оценке.

📐А.7 Математическая модель оценки риска

Система использует количественную модель оценки риска, реализованную в модуле services/risk_service.py. Модель основана на вероятностном подходе и учитывает эффективность существующих мер контроля.

// Шаг 1: Потенциальность события (вероятность реализации угрозы с учётом уязвимости)
P_event = max(P + V − 1, 0)
// где P — вероятность угрозы ∈ [0, 3]; V — уровень уязвимости ∈ [0, 3]
// Интерпретация: P и V задаются экспертно; 1 — низкий, 2 — средний, 3 — высокий уровень
 
// Шаг 2: Начальный (валовый) риск
R_initial = P_event × I
// где I — воздействие (impact) ∈ [0, 9]; R_initial используется как числовой показатель риска
 
// Шаг 3: Остаточный риск (после применения мер контроля)
R_residual = R_initial × (1 − CE)
// где CE — эффективность мер контроля ∈ [0, 1]
// CE = 0 → полное отсутствие контроля: R_residual = R_initial
// CE = 1 → абсолютный контроль: R_residual = 0
 
// Шаг 4: Классификация уровня риска
Level = "Низкий" если R_residual ≤ 3.9
= "Средний" если 4.0 ≤ R_residual ≤ 6.9
= "Высокий" если R_residual ≥ 7.0
 
// Шаг 5: Снижение риска
ΔR = R_initial − R_residual = R_initial × CE
 
// Шаг 6: Приоритет обработки
Priority = f(ценность актива, снижение риска, стоимость меры)
// score ≥ 8 → HIGH; score ≥ 5 → MEDIUM; иначе → LOW

Обоснование модели

Выбранная модель основана на экспертной количественной оценке риска и адаптирована для процессного контекста CSIRT. Формула P_event = max(P+V−1, 0) используется как свёртка вероятности угрозы и выраженности уязвимости. Пороги классификации (≤3.9, 4.0–6.9, ≥7.0) задают управленческие зоны обработки риска. Коэффициент контроля CE (Control Effectiveness) отражает ожидаемое снижение риска после применения организационных или технических мер.

🤖А.8 Rule-based модуль рекомендаций (ai/risk_recommender.py)

Модуль реализован как отдельный сервисный слой с экспертными правилами. Он не является обучаемой ML/LLM-моделью; его задача — формализовать типовые управленческие рекомендации и обеспечить воспроизводимую поддержку принятия решений по рискам.

📝А.12 Модуль аудита и мониторинга качества данных

Журнал аудита (audit_log)

Система автоматически записывает все события изменения данных в таблицу audit_log. Каждая запись включает:

  • user_id, username, role — кто выполнил действие
  • action — тип: create / update / delete / workflow
  • entity_type, entity_id — что было изменено
  • summary — краткое описание
  • before_data (JSON) — состояние до изменения
  • after_data (JSON) — состояние после изменения
  • created_at — временная метка

В текущей БД: 43 записи аудита. Доступно только администратору через /audit_log.

Контроль качества данных (data_quality)

Страница /data_quality выполняет автоматические проверки базы данных:

  • Пробелы без описания (пустое risk_description)
  • Пробелы без назначенного ответственного (risk_owner)
  • Просроченные пробелы (due_date в прошлом, статус не Closed)
  • Пробелы без AI-рекомендации
  • Процессы без подпроцессов
  • Процессы без BPMN-диаграммы
  • Пробелы в статусе Draft дольше 30 дней
  • Пробелы с нулевой эффективностью контроля

Каждая проверка содержит ссылку на соответствующий раздел для исправления.

🏗Б.7 Архитектура системы RiskFlow AI — детальное описание

Система построена по классической трёхуровневой архитектуре (3-Tier Architecture): уровень представления, уровень бизнес-логики и уровень данных. Внутри уровня бизнес-логики дополнительно выделены слой аутентификации/авторизации и сервисный слой. Ниже представлена детальная SVG-диаграмма всех компонентов и потоков данных.

Описание архитектурных уровней

Уровень 0 — Клиентская среда

Взаимодействие с системой осуществляется через стандартный веб-браузер без необходимости установки клиентского ПО. Пользовательский интерфейс реализован средствами HTML5/CSS3 с применением утилитарного CSS-фреймворка Tailwind CSS, загружаемого с CDN (cdn.jsdelivr.net). Для интерактивной визуализации данных используется библиотека Chart.js 4.4 — также подключается через CDN. Коммуникация с сервером осуществляется по протоколу HTTP 1.1 методами GET (загрузка страниц) и POST (отправка форм). Отдельные компоненты используют localStorage для сохранения пользовательских предпочтений интерфейса (например, состояние боковой навигации).

Уровень 1 — Представление (47 шаблонов Jinja2)

Уровень представления реализован на шаблонизаторе Jinja2 (входит в состав Flask). Все шаблоны наследуют базовый макет templates/index.html, который содержит боковую навигацию, мета-информацию, flash-сообщения и JavaScript-обработчики. Каждая страница переопределяет блок {% block content %}. Шаблоны разбиты на 6 категорий: базовый макет, формы рисков и процессов, аналитика, справочная документация, администрирование и аутентификация. Jinja2 обеспечивает server-side rendering — HTML генерируется на сервере и передаётся клиенту в готовом виде (нет SPA/CSR).

Уровень 2 — Аутентификация и RBAC

Подсистема авторизации реализована через Python-декораторы Flask. Пароли хранятся исключительно в виде хэшей PBKDF2-SHA256 (через werkzeug.security). Сессии пользователей реализованы средствами Flask session (подписанные Cookie с секретным ключом). Четыре декоратора образуют иерархию ограничений: @login_required → @expert_required → @risk_editor_required → @admin_required. Дополнительно применяются функции append_process_visibility_filter() и append_process_risk_visibility_filter(), которые динамически добавляют WHERE-условия в SQL-запросы для ограничения видимости данных process_owner.

Уровень 3 — Бизнес-логика (app.py, ~3500 строк)

Основной файл приложения содержит 50 HTTP-маршрутов, организованных в 8 функциональных модулей: организации (4 маршрута), процессы (9), реестр пробелов (6), workflow (1+history), BPMN (2 API), compliance (3), аналитика (5+), администрирование (8+). Workflow-движок реализован в виде словаря PROCESS_RISK_WORKFLOW_ACTIONS, определяющего разрешённые переходы и роли для каждого действия. Функция log_audit_event() автоматически вызывается при каждой операции CUD для записи в audit_log с сериализацией состояния до и после изменения в JSON.

Уровень 4 — Сервисный слой

Сервисный слой вынесен в отдельные модули Python, обеспечивая разделение бизнес-логики расчётов от транспортного уровня Flask. services/risk_service.py содержит три чистые функции без побочных эффектов: calculate_event_potentiality(P, V) реализует формулу max(P+V−1, 0); calculate_process_risk_metrics() выполняет полный цикл расчёта метрик; classify_numeric_risk() классифицирует числовое значение по трём уровням. Модуль ai/risk_recommender.py реализует алгоритм генерации рекомендаций на основе шаблонных правил (rule-based), анализируя комбинации risk_category, P, V, I и CE для формирования текстовых рекомендаций на русском языке.

Уровень 5 — Данные (SQLite, 25 таблиц)

Слой данных реализован на встроенной СУБД SQLite 3.x. Соединение устанавливается через стандартную библиотеку sqlite3 Python. Используется паттерн row_factory = sqlite3.Row для доступа к колонкам по имени. Инициализация схемы выполняется через функцию init_db(), вызываемую при старте приложения. Добавление новых столбцов в существующие таблицы реализовано через ensure_column() — это позволяет безопасно обновлять схему без потери данных. Все 25 таблиц объединены в 5 групп: организации+процессы (5), пробелы (3), рекомендации+auth (4), compliance (3), legacy/справочники (11). Текущий объём базы: 22 пробела, 7 процессов, 4 организации, 33 пользователя, 43 записи аудита.
