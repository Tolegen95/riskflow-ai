# Инструкция для вас (администратора пилота)

Процесс для пилота выбран и подготовлен: **Employee Offboarding (отзыв доступа
увольняющегося сотрудника)**. Он не пересекается с тремя кейсами статьи, и
почти любой ИТ/ИБ/HR/compliance-специалист имеет о нём реальное представление
— поэтому он подходит для пилота.

## Что в этой папке

- `00_process_description.md` — описание процесса и 5 подпроцессов. Дайте это каждому эксперту вместе с `EXPERT_INSTRUCTIONS_RU.md`.
- `EXPERT_INSTRUCTIONS_RU.md` — краткая инструкция для самих экспертов: зачем проводится пилот, почему нельзя обсуждать ответы, как пройти Task 1/2/3, и как сохраняется конфиденциальность. Дайте это каждому эксперту вместе с описанием процесса.
- `EXPERT_PACKET_print_this.md` — **главный документ**: распечатайте по одной копии на каждого эксперта. Внутри уже вписаны названия подпроцессов, шкала-рубрика и формы для Task 1 и Task 2.
- `expert_vote_task1_task2_RU.html` — основная русская веб-страница для голосования экспертов по Task 1/2; открывается двойным кликом, сервер не нужен.
- `task3_sus_questionnaire.md` — анкета usability, выдаётся ПОСЛЕ Task 1/2, когда эксперт увидит реальный расчёт системы.
- `ground_truth_scoring_rationale.md` — **не показывать экспертам**. Это "официальные" P/V/I/CE, которые я заложил как ввод процесса, и посчитанные через реальные формулы системы (`services/risk_service.py`) итоговые категории.
- `framework_categories.csv` — уже готов, содержит реальные посчитанные категории (нужен для сравнения с холистической оценкой экспертов).
- `task1_framework_scoring_TOFILL.csv`, `task2_holistic_rating_TOFILL.csv`, `task3_sus_TOFILL.csv` — сюда вы вручную переносите то, что эксперты заполнили на бумаге (или экране).

## Два способа собрать ответы

**Вариант А — бумага:** раздайте `EXPERT_INSTRUCTIONS_RU.md` +
`00_process_description.md` + `EXPERT_PACKET_print_this.md`, соберите бланки,
вручную впишите цифры в `_TOFILL.csv`.

**Вариант Б — веб-форма (быстрее, меньше ошибок ввода):** отправьте эксперту
файлы `EXPERT_INSTRUCTIONS_RU.md`, `00_process_description.md` и
`expert_vote_task1_task2_RU.html` (форма просто открывается двойным кликом в браузере,
сервер не нужен). Эксперт вводит свой Expert ID, заполняет Section A, жмёт
"Заблокировать Section A и перейти к Section B", заполняет Section B, жмёт
"Скачать мои ответы Task 1 и Task 2" — браузер сохранит
`task1_<ID>.csv` и `task2_<ID>.csv`. Эксперт присылает вам эти 2 файла.
Аналогично `web_form_task3_sus.html` для Task 3 (отдельно, после показа расчёта).
Когда соберёте файлы от всех экспертов в одну папку — объедините одной командой:
```bash
../.venv/bin/python merge_responses.py --task1 "responses/task1_*.csv" --task2 "responses/task2_*.csv" --task3 "responses/task3_*.csv"
```
Скрипт сам проверит, что нет пустых полей и что названия подпроцессов совпадают
дословно — если эксперт что-то не заполнил или переименовал подпроцесс, скрипт
откажется объединять и укажет, в каком файле ошибка.

## Шаги

1. **Наберите 5+ экспертов.** В бумажных шаблонах уже проставлены ID `P1`...`P5` — если экспертов больше, просто добавьте строки `P6`, `P7` и т.д. (для веб-формы просто раздайте разные ID).

2. **Раздайте `EXPERT_INSTRUCTIONS_RU.md` + `00_process_description.md`** каждому эксперту, затем форму (бумажную `EXPERT_PACKET_print_this.md` или веб-страницу `expert_vote_task1_task2_RU.html`) — вариант А или Б выше. Важно: эксперты должны заполнять **независимо друг от друга**, не видя чужие ответы и не обсуждая между собой до того, как все закончат.

3. **Соберите ответы**: вариант А — вручную впишите в `task1_framework_scoring_TOFILL.csv` / `task2_holistic_rating_TOFILL.csv`; вариант Б — сложите присланные CSV в одну папку и запустите `merge_responses.py` (см. выше), затем переименуйте `*_MERGED.csv` в `*_TOFILL.csv` либо укажите `_MERGED.csv` напрямую в команде анализа на шаге 6.

4. **Покажите каждому эксперту реальный расчёт системы** для этого процесса (введите процесс в приложение как обычно через админку — 5 подпроцессов, значения из `ground_truth_scoring_rationale.md` — либо просто покажите готовую таблицу оттуда) и **только после этого** дайте Task 3 (бумажный `task3_sus_questionnaire.md` или `web_form_task3_sus.html`).

5. **Перенесите/объедините ответы Task 3** в `task3_sus_TOFILL.csv` (или используйте `*_MERGED.csv`).

6. **Запустите анализ:**
   ```bash
   cd validation_study
   ../.venv/bin/python analyze_validation.py \
     --task1 PILOT_employee_offboarding/task1_framework_scoring_TOFILL.csv \
     --task2 PILOT_employee_offboarding/task2_holistic_rating_TOFILL.csv \
     --task3 PILOT_employee_offboarding/task3_sus_TOFILL.csv \
     --framework-categories PILOT_employee_offboarding/framework_categories.csv
   ```

7. **Пришлите мне вывод скрипта** (или просто скажите "готово, вот реальные данные") — я перепишу Section 6.1 статьи с настоящими числами вместо "not yet executed", и это уже будет честный, настоящий результат, каким бы он ни оказался.

## Важно

Если экспертов будет не 5, а, скажем, 3 или 7 — ничего страшного, просто
поправьте число строк в `_TOFILL.csv` файлах перед запуском анализа. Слабый
или неоднозначный результат (низкое согласие экспертов, несовпадение с
категорией системы) — это тоже честный и полезный результат для статьи, не
повод скрывать пилот или подгонять числа.
