# Pilot Process: Employee Offboarding (Access Revocation)

Give this page to every expert before Task 1. It is the only context they
should need — do not add your own organization's specifics unless every
expert works from the same added detail, or you reintroduce the exact
inconsistency the study is trying to measure.

## Why this process

It was chosen deliberately instead of reusing the paper's three illustrative
case studies (information-security incident management, SCADA monitoring,
personal-data processing), so that the pilot is not run on a case the
framework's thresholds were implicitly shaped around. Employee offboarding is
also something almost any IT, security, HR, or compliance professional has
direct working knowledge of, which matters for a rating task to be meaningful
— experts need enough real familiarity with the process to have a genuine,
independent opinion, not just a plausible guess from the written description
alone.

**Methodological note for the paper (Section 6.1.1):** this is a *vignette-based*
design — participants rate a standardized written case description rather than
their own organization's live process. This is a well-established design in
inter-rater reliability research (e.g., clinical vignette studies), and it is
what makes it possible to run the exact same instrument across experts from
different organizations. It is a legitimate but different claim from an
in-situ organizational assessment; report it as such rather than implying
participants audited their own live process.

## Process description

**Name:** Employee Offboarding (Access Revocation)
**Owner role:** IT Security Manager
**Trigger:** An employee resigns, is terminated, or changes role in a way that
requires removing some or all of their current system access.
**Input:** Offboarding notice (resignation letter, termination notice, or role-change request).
**Output:** Fully revoked access, returned assets, completed handover, and an archived compliance record.

## Subprocesses

1. **Offboarding request submission** — HR or the departing employee's manager
   formally initiates the offboarding request in the HR system, recording the
   employee's last working day and the reason for departure.

2. **Access inventory and revocation** — IT/security identifies every system,
   application, and physical access credential the employee holds (including
   SaaS tools, shared accounts, and remote access), and revokes or disables
   each one.

3. **Asset return and verification** — the employee returns company-owned
   devices, badges, tokens, and any other physical assets, checked against an
   asset registry.

4. **Knowledge transfer and handover** — the departing employee's manager
   ensures critical knowledge, active tasks, and any shared credentials or
   system ownerships are transferred to a successor or team member.

5. **Compliance archival** — HR/compliance archives the offboarding record
   (dates, approvals, evidence that access was revoked) according to the
   organization's retention policy.
